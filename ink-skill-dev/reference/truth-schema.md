# truth-schema — 7 个真相文件的 schema 参考

> 本模块由 ink.skill 在任何需要读写 truth files 时 Read。
> 真相文件：character_matrix / worldbook / particle_ledger / chapter_summaries / current_state / foreshadow / timeline。

---

# 真相文件 Schema 规范

每本书的 `story/` 目录下有 7 个真相文件（truth files），是世界状态的持久化层。它们的 schema 和合并语义各不相同，**搞混了会引发数据丢失**。

本章节定义所有 7 个文件的精确 schema、合并 key、更新规则。写章、审计、修订、结算都必须严格遵守。

---

## 总表：7 个 truth files 速查

| 文件 | 中文名 | 格式 | 合并 key | 主要更新方 |
|------|-------|------|---------|-----------|
| `current_state.md` | 当前状态卡 | 字段键值表 | 整体覆盖 | Settler（每章覆盖） |
| `particle_ledger.md` | 资源账本 | 7 列表格 | `[6]` 事件ID | Writer/Settler/Analyzer |
| `pending_hooks.md` | 伏笔池 | 7 列表格 | `[0]` hook_id | Writer/Settler/Analyzer |
| `subplot_board.md` | 支线进度板 | 9 列表格 | `[0]` 支线ID | Settler/Analyzer |
| `emotional_arcs.md` | 情感弧线 | 6 列表格 | `[0, 1]` 角色+章节 | Settler/Analyzer |
| `character_matrix.md` | 角色交互矩阵 | 3 子表 | 每子表独立 | Settler/Analyzer |
| `chapter_summaries.md` | 章节摘要 | 8 列表格 | `[0]` 章节 | Settler（追加） |

## 1. current_state.md（当前状态卡）

### 格式

字段键值表：

```markdown
# 当前状态

| 字段 | 值 |
| --- | --- |
| 当前章节 | 5 |
| 当前位置 | <地点> |
| 主角状态 | <身体/心理状态简述> |
| 当前目标 | <本阶段主角要达成的目标> |
| 当前限制 | <当前的约束条件，如资源不足/身份隐藏等> |
| 当前敌我 | <势力 A（敌对）/ 势力 B（同盟）/ 势力 C（中立）> |
| 当前冲突 | <本章或本阶段的核心冲突> |
```

### 更新规则

- **每章写完后整体覆盖**（不是追加）
- 字段固定为 7 行：当前章节、当前位置、主角状态、当前目标、当前限制、当前敌我、当前冲突
- 值字段内容可以多行（单元格内换行用 `<br>` 或保持简洁）
- **合并 key：无**（整体覆盖）

### 快照语义

- `snapshots/N/current_state.md` = 写完第 N 章后的状态卡
- **绝不用最新状态卡覆盖旧快照**

## 2. particle_ledger.md（资源账本）

### 格式

7 列表格：

```markdown
# 资源账本

| 章节 | 资源名称 | 期初 | 变动 | 期末 | 事由 | 事件ID |
|------|----------|------|------|------|------|--------|
| 0 | - | 0 | 0 | 0 | 开书初始 | init-0 |
| 4 | 情报权 | 27 | +1 | 28 | <事由说明> | ch4-情报权-1 |
| 4 | 情报权 | 28 | +1 | 29 | <事由说明> | ch4-情报权-2 |
| 4 | 情报权 | 29 | +1 | 30 | <事由说明> | ch4-情报权-3 |
```

**表头严禁修改**。列名、列数必须严格一致。

### 事件ID 规则（极其重要）

1. **格式**：`ch{章节号}-{资源名}-{序号}`，例如 `ch3-情报-2`。资源名允许中文，序号从 1 开始。

2. **初始化行**：统一用 `init-0`。

3. **同一 (章节, 资源) 内多个独立事件**：序号递增
   - `ch3-情报-1`、`ch3-情报-2`、`ch3-情报-3`

4. **修正已有事件**：**必须复用**原事件ID，不要生成新的。这是合并去重的唯一判据。

5. **新增事件**：如果这是全新事件（当前账本里不存在），生成一个未使用的新 事件ID。

6. **验算铁律**：期初 + 变动 = 期末，三项必须可验算。

### 合并 key

`[6]` = 事件ID 列（第 7 列，0 索引为 6）。

**合并逻辑**：
- 事件ID 相同 → 替换整行（修正场景）
- 事件ID 不同 → 追加到表末（新增场景）

### Sentinel（未更新哨兵）

如果本章没有任何资源变动，Settler 可以输出：

```
☆ 资源账本无变动 ☆
```

或：

```
(资源账本未更新)
```

检测到 sentinel 时，**不要覆盖**现有 `particle_ledger.md`。

## 3. pending_hooks.md（伏笔池）

### 格式

7 列表格：

```markdown
# 伏笔池

| hook_id | 起始章节 | 类型 | 状态 | 最近推进 | 预期回收 | 备注 |
|---------|----------|------|------|----------|----------|------|
| mentor-oath | 3 | relationship | progressing | 7 | 揭开师徒旧约真相 | ch7 配角 B 提到旧誓 |
| lost-key | 1 | mystery | progressing | 5 | 解开信物来历 | ch5 主角发现第二件信物 |
```

### hook_id 规则

- 格式：`{主题}-{关键词}`，英文小写加连字符，简短有辨识度
- 示例：`mentor-oath`、`lock-sigil`、`missing-ledger`
- **hook_id 一经创建永不修改**，这是合并的唯一判据

### 状态枚举

| 状态 | 说明 |
|------|------|
| `progressing` | 推进中 |
| `mentioned` | 本章只是被提到，未推进 |
| `resolved` | 已回收 |
| `deferred` | 延后（超过 5 章未推进） |

### 字段语义

- **起始章节**：伏笔首次埋下的章节号
- **最近推进**：最后一次真正推进的章节号（**只是被提到不算推进**）
- **预期回收**：预期在哪里收回，用短句描述
- **备注**：本次更新的简短说明

### 伏笔追踪铁律（必须严格执行）

1. **新伏笔**：只有当正文中出现会延续到后续章节且有具体回收方向的未解问题时，才新增 hook_id。**不要为旧 hook 的换说法再开新 hook**。

2. **提及伏笔**：已有伏笔在本章被提到但没有新增信息 → 状态可以改为 `mentioned`，**不要更新"最近推进"**。

3. **推进伏笔**：已有伏笔出现了新事实、证据、关系变化、风险升级或范围收缩 → **必须更新"最近推进"为当前章节号**，更新状态和备注。

4. **回收伏笔**：伏笔在本章被明确揭示、解决、或不再成立 → 状态改为 `resolved`，备注回收方式。

5. **延后伏笔**：超过 5 章未推进 → 状态改为 `deferred`，备注原因。

6. **铁律**：不要把"再次提到""换个说法重述"当成推进。只有状态真的变了，才更新"最近推进"。

### 合并 key

`[0]` = hook_id 列（第 1 列）。

### Sentinel

```
☆ 伏笔池无变动 ☆
```

## 4. subplot_board.md（支线进度板）

### 格式

9 列表格：

```markdown
# 支线进度板

| 支线ID | 支线名 | 相关角色 | 起始章 | 最近活跃章 | 距今章数 | 状态 | 进度概述 | 回收ETA |
|--------|--------|----------|--------|------------|----------|------|----------|---------|
| SL-01 | <支线名> | 主角/配角B | 2 | 7 | 0 | active | <进度概述> | ch9-11 |
| SL-02 | <支线名> | 主角/导师 | 3 | 5 | 2 | paused | <进度概述> | ch15+ |
```

### 支线ID 规则

- 格式：`SL-{两位数序号}`，如 `SL-01`、`SL-02`
- 一经创建永不修改

### 状态枚举

| 状态 | 说明 |
|------|------|
| `active` | 活跃推进中 |
| `paused` | 暂停（有意识的搁置）|
| `stalled` | 停滞（超过 5 章未动）|
| `resolved` | 已完结 |
| `abandoned` | 放弃 |

### 字段语义

- **距今章数** = 当前章节号 - 最近活跃章
- **回收ETA**：预期在哪几章范围内完结

### 合并 key

`[0]` = 支线ID 列（第 1 列）。

### Sentinel

```
☆ 支线进度板无变动 ☆
```

## 5. emotional_arcs.md（情感弧线）

### 格式

6 列表格：

```markdown
# 情感弧线

| 角色 | 章节 | 情绪状态 | 触发事件 | 强度(1-10) | 弧线方向 |
|------|------|----------|----------|------------|----------|
| 主角 | 5 | 戒备/隐忍 | <触发事件> | 7 | 压抑上升 |
| 主角 | 7 | 冷怒/决断 | <触发事件> | 8 | 压抑上升 |
| 配角B | 7 | 焦灼/试探 | <触发事件> | 6 | 摇摆 |
```

### 字段语义

- **情绪状态**：简洁标签化（2-4 字），不要写成句子
- **强度**：1-10 整数
- **弧线方向**：描述趋势（如"压抑上升"、"释放下降"、"摇摆"、"平稳"）

### 合并 key

`[0, 1]` = 角色 + 章节（前两列）。

**合并逻辑**：
- 同一 (角色, 章节) 已存在 → 替换该行
- 不同 (角色, 章节) → 追加

### Sentinel

```
☆ 情感弧线无变动 ☆
```

## 6. character_matrix.md（角色交互矩阵）

### 格式（3 子表结构）

**必须是 3 个 `### ` 子表**，顺序固定：

```markdown
# 角色交互矩阵

### 角色档案

| 角色 | 核心标签 | 反差细节 | 说话风格 | 性格底色 | 与主角关系 | 核心动机 | 当前目标 |
|------|----------|----------|----------|----------|------------|----------|----------|

（每行一个角色。Section 0 的 key 是第 0 列"角色"。）

### 相遇记录

| 角色A | 角色B | 首次相遇章 | 最近交互章 | 关系性质 | 关系变化 |
|-------|-------|------------|------------|----------|----------|

（每行一对角色。Section 1 的 key 是前两列 [角色A, 角色B]。不要为从未交互的角色对创建行。）

### 信息边界

| 角色 | 已知信息 | 未知信息 | 信息来源章 |
|------|----------|----------|------------|

（每行一条"某角色知道/不知道某关键信息"的记录。Section 2 的 key 是第 0 列 + 第 3 列 [角色, 信息来源章]。）
```

### 合并 key（按子表分别）

| 子表 | key 列 | 说明 |
|------|--------|------|
| 角色档案 | `[0]` 角色 | 同一角色已存在 → 替换 |
| 相遇记录 | `[0, 1]` 角色A + 角色B | 同一对角色已存在 → 替换 |
| 信息边界 | `[0, 3]` 角色 + 信息来源章 | 同一 (角色, 来源章) 已存在 → 替换 |

### 扁平单表 fallback

如果现有 `character_matrix.md` 是扁平单表（没有 `### ` 子表），视为旧版 schema：

- 两边都是扁平 → 按 `[0]` 角色合并
- 旧的扁平 + 新的 3 子表 → **升级为 3 子表**（接受新格式）
- 旧的 3 子表 + 新的扁平 → **保留 3 子表**（拒绝回退）
- 两边都是 3 子表 → 按每子表 key 合并

### Sentinel

```
☆ 角色交互矩阵无变动 ☆
```

## 7. chapter_summaries.md（章节摘要）

### 格式

8 列表格：

```markdown
# 章节摘要

| 章节 | 标题 | 出场人物 | 关键事件 | 状态变化 | 伏笔动态 | 情绪基调 | 章节类型 |
|------|------|----------|----------|----------|----------|----------|----------|
| 1 | <标题> | 主角 | <关键事件概述> | <状态变化> | 埋 lost-key | 迷茫/警觉 | 开篇 |
| 7 | <标题> | 主角/配角B/配角C | <关键事件概述> | 情报权 30 | 推进 SL-01 | 冷怒/决断 | 高潮过渡 |
```

### 字段语义

- **出场人物**：本章有台词或关键行动的角色，用 `/` 分隔
- **关键事件**：一句话概括本章核心事件
- **状态变化**：本章末尾的主角状态变化
- **伏笔动态**：新增/推进/回收的 hook_id（简写）
- **情绪基调**：本章整体情绪氛围
- **章节类型**：开篇 / 过渡 / 高潮 / 转折 / 回忆 等

### 合并 key

`[0]` = 章节列。

**合并逻辑**：同一章节号已存在 → 替换（修订场景）；新章节 → 追加。

### Sentinel

```
☆ 章节摘要无变动 ☆
```

## 通用合并原则（所有 truth files 共用）

### 什么时候由 LLM 直接更新 vs 调用 merge-truth.py

| 场景 | 方式 |
|------|------|
| **写章后结算**（日常）| LLM 直接更新：读现有文件 → 应用变更 → 写回 |
| **修订后同步**（日常）| LLM 直接更新 |
| **从快照恢复后重建** | 调用 `merge-truth.py` |
| **批量 rebuild（重建 ledger/hooks）** | 调用 `merge-truth.py` |
| **导入外部章节** | 调用 `merge-truth.py` |

**原则**：LLM 自己产出的变更由 LLM 自己更新；外部来源或批量操作用脚本。

### LLM 更新 truth file 的步骤

1. 读取现有文件的完整内容
2. 确定需要新增/修改的行（基于合并 key 判断）
3. 做定点修改（只改变化的行），或写回完整更新后的文件
4. 更新后校验：检查关键数据是否还在（**行数不应减少**，除非用户明确要求）

### 永远不要做的事

- ❌ 删除 truth file 的已有数据行
- ❌ 修改表头（列名、列数、顺序）
- ❌ 改变 sentinel 的确切字符串（否则检测失败）
- ❌ 在 snapshots/N/ 下写入任何文件
- ❌ 在没有读取现有文件的情况下直接整体重写（会丢数据）

### Schema 守卫（自检）

写回 truth file 前，快速检查：

1. 表头是否与本文档定义一致？（列名、列数）
2. 行数是否 ≥ 原文件？
3. 所有 key 列是否非空？
4. 对于 ledger：期初 + 变动 = 期末？
5. 对于 character_matrix：是否保留了 3 个 `### ` 子表结构？

任一检查失败 → 不要写回，报告错误。
